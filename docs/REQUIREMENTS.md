## 📌 使用说明
- **不要自己增加待开发和待修复的内容**：待开发和待修复内容应该严格控制，不能由AI增加
- **执行中任务**：已经设计等待确认或需求需要确认的任务，移到待开发板块列出，在待开发板块收到指令消息后才去执行
- **新增需求/BUG**：直接在对应分类下追加条目，无需任何标记
- **已完成条目**：由 AI 完成（需求和BUG）后会在状态列打上 `✅ 已完成` 标记,并移动到已完成板块（已完成和已修复）
- **优先级**：🔴 紧急 | 🟡 中等 | 🟢 低优先级
- **类型**：`[需求]` `[BUG]` `[优化]` `[文档]`
- **必做**：总结任务发送飞书、在最后全部完成后更新本地前后端并启动，记住我现在在**windows环境**中
- **执行**：阅读完文档就开始工作，不要来询问我是否继续

### MCP SSE 会话过期问题分析（v1.3.6）

**问题现象**：
```
[warning] 404 status sending message: {"error":"会话不存在或已过期，请重新建立 SSE 连接"}
```

**根因分析**：
- 后端 SSE 会话存储在内存中（Map<sessionId, session>）
- 后端重启或服务中断会清除所有会话
- 原心跳间隔 25 秒，某些网络环境（特别是有代理/防火墙）的超时限制更短（10-20s）
- 网络长连接可能被中间节点主动断开）

**改进方案**（已更新至 v1.3.19）：
1. **心跳频率增加**：从 25 秒改为 15 秒，更激进地保活连接
2. **Streamable HTTP 支持**：同一 `/api/mcp/sse` 地址支持 `POST` 会话模式，404 后由 VS Code 自动新建会话
3. **Legacy SSE 回退保留**：旧客户端仍可通过 `GET /api/mcp/sse` + `POST /api/mcp/message` 工作
4. **移除冗余空闲过期**：删除 60 秒无活动强制断开，避免把正常长连接误表述为“会话超时”
5. **错误日志详化**：记录会话过期和传输模式，便于区分 HTTP 会话丢失与 legacy SSE 断线

**后续优化建议**：
- 考虑支持 Webhook 重试机制（当飞书 API 不可用时自动重试）
- 集成页面增加"最后触发时间"列，方便排查
- 管理员可查看所有用户的机器人（当前只能看自己的）
- 支持批量导出通知历史为 CSV
